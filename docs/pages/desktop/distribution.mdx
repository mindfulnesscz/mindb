# Package collection and local publishing

## Package distribution

The collect/distribute stage finds package folders by `packagePrefix` only (e.g. `📦`). Migrated `Name __hash` asset folders are not anchors.

For each package it collects the **highest version only** from every `OUT` under the package’s parent tree (siblings and nested assets), then hard-mirrors those files into the package. Orphans and thumbnails are deleted from the package (never `🚫`-disconnected). Other package folders are not walked, so anchors don’t pull from each other.

## Local target publishing

Publish always exports **highest version only**. With `includePackages`, packages are discovered the same way, refreshed/selected from OUT, then copied into the target at nested source-relative paths. Target package folders are then **hard-purged** to match live files only (no older versions or renamed-tag leftovers; no `🚫` inside packages).

Outside packages, previously published items missing from the current export are renamed on the **target** with a `🚫` prefix rather than deleted.

Use local publishing for a filesystem-synced SharePoint/OneDrive tree when folder structure is important. Use [cloud publishing](/cloud-storage/publishing) when flat per-file API delivery and share-link collection are preferred.
