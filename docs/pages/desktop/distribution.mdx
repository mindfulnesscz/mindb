# Package collection and local publishing

## Package distribution

The collect/distribute stage finds folders beginning with `packagePrefix`. For each package it inspects sibling folders, prefers their configured `OUT`, and also recognizes direct “orphan” files. Selected files are translated through the naming vocabulary and copied into the package.

The unchanged test compares destination mtime with source mtime; if mtimes are unavailable it compares size. A destination at least as new is skipped. This is fast but not cryptographic: equal size is not proof of equal content.

Stable-identity migration removes the package prefix and appends ` __<8 hex>`. Such a folder is no longer discovered by prefix-based package collection, a known design limitation.

## Local target publishing

The publish stage mirrors the relevant source hierarchy into `targetFolder`, translating published filenames and removing stable-ID suffixes from destination folder names. Previously published items missing from the current source are renamed with a `🚫` prefix rather than deleted, making disconnection visible and reversible.

Use local publishing for a filesystem-synced SharePoint/OneDrive tree when folder structure is important. Use [cloud publishing](/cloud-storage/publishing) when flat per-file API delivery and share-link collection are preferred.
