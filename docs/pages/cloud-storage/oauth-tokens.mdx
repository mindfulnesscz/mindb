# OAuth and token lifecycle

Access tokens authorize provider API calls; refresh tokens obtain new access tokens without repeating consent.

**Destination structure** (name, type, remote path, `minRole`, generate-link, etc.) is edited in the **web portal** and pulled into desktop via Sync. **Tokens and Google client secret** stay on the workstation. Before any definition is written to Supabase from older desktop paths, tokens are stripped — credentials must remain machine-local.

Status is purely time-based: no token, fresh, expiring within one hour, or expired. Google access tokens typically have a one-hour lifetime, so “Expires soon” can appear immediately. The pipeline filters for the presence of an access-token string, not current expiry, and does **not** automatically refresh before upload. Use Refresh token in Settings before a run.

Refresh behavior:

- Dropbox: refresh-token POST with app key.
- OneDrive: native refresh call with client ID and tenant authority; Microsoft may rotate the refresh token.
- Google: token POST with client ID, client secret, and refresh token.

Revocation, password/policy changes, consent removal, Testing-mode expiry, or missing offline consent can invalidate refresh tokens. A failed refresh requires reconnecting. Protect client configuration backups — they may still contain live **provider OAuth** tokens. Service-role and permanent R2 secrets should not be present in current exports.
