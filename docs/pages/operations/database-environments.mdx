# Database environments

The database schema is code: it lives in `supabase/migrations/` at the repo root, replayable from zero, and no SQL reaches production that didn't first exist as a migration file. The days of "run once in the SQL editor" are over — that practice produced silent drift (a portal download button reading a column the live DB wiped, functions that existed nowhere in source).

## The three tiers

| Tier | What | Schema arrives by |
|---|---|---|
| Local | Docker stack via `npm run db:start` | `npm run db:reset` — replays all migrations + `seed.sql` |
| Staging | Hosted Supabase project | CI (`db.yml`) on push to the **`staging`** branch |
| Production | Live client-facing project | CI on push to **`main`** (approval-gated) |

Day-to-day feature/pipeline work should use **shared Staging** for both Supabase and R2 — see [Team staging setup](/getting-started/team-dev-setup). Local is for schema/migration checking, not hybrid rehearsals against a shared bucket.

## Local development

### One-time setup: Docker + Supabase CLI

The local stack runs as Docker containers, so a container runtime must be installed and *running* before any `db:*` command.

1. **Install Docker Desktop** from [docker.com](https://www.docker.com/products/docker-desktop/) (drag to Applications), or `brew install --cask docker-desktop`. OrbStack is a lighter alternative if you prefer (`brew install --cask orbstack`) — either provides the same `docker` daemon; install exactly one.
2. **Launch it once** (`open -a Docker`) and complete the first-run dialog. Enable *Start Docker Desktop when you sign in* (Settings → General) so the daemon is always available.
3. **Verify the daemon**: `docker info` should print a server version, not a connection error. If `docker` isn't found, add `/usr/local/bin` to your `PATH` — Docker Desktop symlinks its CLI there.
4. **Resources** (Settings → Resources): the defaults are fine; the Supabase stack needs roughly 2 GB RAM and ~3 GB disk for images.
5. **Install the Supabase CLI**: `brew install supabase/tap/supabase`.

### Daily commands

```bash
npm run db:start    # boots Postgres/Auth/PostgREST/Studio locally (first run pulls images)
npm run db:reset    # rebuild from migrations + seed — your DB is disposable
npm run db:stop     # free the resources when done
npm run db:types    # regenerate database.types.ts from the local schema
```

`db:start` prints the local URLs and keys when ready — API `http://127.0.0.1:54321`, Studio `http://localhost:54323`, plus the `anon` and `service_role` keys (fixed, well-known local dev values; safe to commit to `.env.local`).

### Seed data and sign-ins

`npm run db:reset` loads `supabase/seed.sql`: the **Acme Studio** demo client (slug `acme`, whitelisted domain `acme.test`), a small taxonomy, four assets covering the interesting shapes (public/client/internal perms, a variant pair, version history, view/download events) — **and a ready-made admin**:

| | |
|---|---|
| Email | `admin@acme.test` |
| Password | `dchub-local` |
| Role | `admin` — full portal plus the admin area |

It's recreated by every `db:reset`, so it always exists on a fresh local stack. (Seeding a user by inserting into `auth.users` is a local-only pattern — never replicate it on staging or production.)

**Two sign-in flows, one inbox rule:** the portal's email/OTP flow sends real emails — locally those are **not delivered anywhere**; they land in **Mailpit at `http://localhost:54324`**. Open Mailpit, click the latest message, use the link/code. Password sign-in (where offered) works directly with the credentials above.

**Creating additional test users** (e.g. to see the member experience):

1. Studio (`http://localhost:54323`) → **Authentication** → **Add user** → *Create new user*.
2. Use any `@acme.test` address and a password; tick *Auto Confirm User*.
3. Done — the `handle_new_user` trigger sees the whitelisted domain and creates a `profiles` row with `role = 'member'` attached to Acme Studio. Members get the client portal; the **admin area refuses them** by design.
4. Prefer promoting via web **Users** tab (`update_user_access`) on Staging/local once signed in as admin. SQL fallback:

   ```sql
   update public.profiles set role = 'editor'   -- or 'admin' or 'member'
   where id = (select id from auth.users where email = 'me@acme.test');
   ```

Note that `db:reset` wipes users along with everything else — manually created users are gone after a reset (the seeded admin comes back automatically).

### Wiring the apps to an environment

- **Web portal** — `web/apps/client-hub/.env.local` (copy from `.env.local.example`):

  ```bash
  VITE_SUPABASE_URL=http://127.0.0.1:54321
  VITE_SUPABASE_ANON_KEY=<anon key printed by db:start>
  ```

  Without these the portal falls back to mock data. `.env.local` targets the local stack; committed mode files switch backends without editing it:

  ```bash
  npm run dev:web            # local stack (.env.local)
  npm run dev:web:staging    # staging (.env.staging)
  npm run dev:web:prod       # production (.env.production) — real client data, look don't touch
  ```

  Mode files override `.env.local` (Vite precedence) and contain only public values. Restart the dev server when changing env files — Vite reads them at startup.

- **Desktop app** — environments are first-class: *Configure server* / Settings → Environment holds each environment's URL and anon key. The desktop holds no service-role or permanent R2 keys; sync and grants run as the signed-in user under RLS. After sign-in it fetches that environment's **client list from the database** (editors: `client_members`; admins: all). Only **folder paths and OAuth tokens** attach per `(environment, client)` in `client-local.json`. Switching environment re-authenticates against the other backend.

### Troubleshooting

- `Cannot connect to the Docker daemon` — Docker Desktop isn't running; `open -a Docker` and wait for the whale icon to settle.
- Port conflict on 54321–54324 — another Supabase project is running; `supabase stop --project-id <other>` or change ports in `supabase/config.toml`.
- Stack in a weird state — `supabase stop --no-backup` then `npm run db:start` gives a clean slate; migrations + seed make the data fully reproducible.
- Sign-in emails never arrive — they're in Mailpit (`http://localhost:54324`), not your inbox.
- Magic link opens `127.0.0.1:3000` and refuses to connect — the auth `site_url` doesn't match the portal's dev port. It's set to `http://localhost:5173` (Vite) in `supabase/config.toml`; config changes need `supabase stop && npm run db:start` to apply. If Vite ever starts on a different port (5174 when 5173 is busy), add it to `additional_redirect_urls`.

## Setting up the staging project (one-time)

1. [supabase.com/dashboard](https://supabase.com/dashboard) → **New project** (e.g. `dc-hub-staging`, region near you). Generate and save the database password.
2. Collect from Settings: the **project ref** (General) and the **anon key** (API). (The service_role key stays server-side — the desktop no longer uses it.)
3. First schema apply — staging is fresh, so unlike production it receives *everything*:

   ```bash
   supabase login
   supabase link --project-ref <staging-ref>
   supabase db push
   ```

4. Hand future pushes to CI (see [CI secrets](#ci-secrets)); after setup, avoid bare `supabase db push` from a workstation — `link` remembers only one project, and CI always links explicitly.

## Making a schema change

1. `supabase migration new <short_name>` — creates a timestamped file in `supabase/migrations/`.
2. Write the SQL. `npm run db:reset` must succeed from zero.
3. `npm run db:types` and commit the regenerated types with the migration.
4. PR → CI validates migrations. Merge to **`staging`** → staging project. Merge to **`main`** → production (approval-gated).

Never edit an already-merged migration; write a new one.

## Baseline and production adoption

`20260711000000_baseline.sql` is the production schema as of v2.3.0, verified against live introspection (it deliberately preserves production quirks; the migration after it fixes the `entities` column type). Production must **not** have the baseline pushed — when linking the prod project for the first time, mark it applied instead:

```bash
supabase link --project-ref <prod-ref>
supabase migration repair --status applied 20260711000000
supabase db push   # applies only migrations newer than the baseline
```

Two functions in the baseline (`get_all_profiles`, `update_user_role`) were reconstructed from portal usage because they existed only in production. After linking, run `supabase db diff --linked` and reconcile any differences with a repair migration.

## CI secrets

`db.yml` needs, in the `staging` / `production` GitHub environments: `SUPABASE_ACCESS_TOKEN` (personal access token), `SUPABASE_STAGING_REF` / `SUPABASE_PROD_REF` (project refs), `SUPABASE_STAGING_DB_PASSWORD` / `SUPABASE_PROD_DB_PASSWORD`. Configure the `production` environment to require manual approval.

## Storage tiers — environment-level R2

R2 follows the same tiers as databases: one bucket + public domain **per environment**, stored as edge function secrets — **not** on `clients` rows. Object keys are prefixed `{client_id}/` (logos: `branding/{client_id}/`). Full operator steps: [R2 CDN publishing](/desktop/cdn).

### Provisioning secrets (per environment)

**Checklist — repeat for every hosted project (staging, production, …):**

| Step | Action | Verify |
|------|--------|--------|
| 1 | Cloudflare: create R2 bucket + public domain (r2.dev or custom) | Bucket exists; CORS allows `GET`/`HEAD` |
| 2 | Cloudflare: create R2 API token (Workers R2 Storage Read+Edit) | Token verify returns success |
| 3 | `supabase link --project-ref <ref>` | `supabase projects list` shows linked project |
| 4 | Set **all five** edge secrets (below) | `supabase secrets list` shows every name |
| 5 | `supabase functions deploy` (or merge to `staging`/`main` and let CI deploy) | Dashboard → `r2-grant` is deployed |
| 6 | Desktop / portal: point environment at this project's URL + anon key | Pipeline CDN grant returns 200, not 503 |

Missing **any** of the five names causes `r2-grant` **503** — common partial setup: only the Cloudflare trio (`CF_*`, `R2_PARENT_*`) without `R2_BUCKET` / `R2_PUBLIC_DOMAIN`.

| Secret | Source |
|--------|--------|
| `R2_BUCKET` | Bucket name (e.g. `dc-hub-staging`) |
| `R2_PUBLIC_DOMAIN` | Public origin, no trailing slash (r2.dev or custom domain) |
| `CF_API_TOKEN` | R2 account API token value |
| `CF_ACCOUNT_ID` | R2 account ID |
| `R2_PARENT_ACCESS_KEY_ID` | Access Key ID from the same token create screen |

```bash
supabase link --project-ref <env-ref>
supabase secrets set R2_BUCKET=… R2_PUBLIC_DOMAIN=… CF_API_TOKEN=… CF_ACCOUNT_ID=… R2_PARENT_ACCESS_KEY_ID=…
supabase secrets list   # names only — values are never shown again
supabase functions deploy
```

**CLI: remote vs local.** `supabase link` only tells the CLI which **hosted** project receives `db push`, `secrets set`, etc. It does **not** stop or replace the local Docker stack. Local database is always separate:

```bash
npm run db:start    # local Postgres/API at 127.0.0.1:54321
npm run db:reset    # replay migrations + seed locally
supabase status     # local URLs and keys (when stack is running)
```

To stop targeting a remote project from this repo checkout: `supabase unlink`. Re-link anytime with `supabase link --project-ref <ref>`. The link is stored under `.supabase/` in the repo (gitignored) — it is per working copy, not global.

For a **new environment end-to-end** (schema, functions, secrets, auth URLs, founding admin), see [Configuration and deployment — Standing up a new environment](/operations/deployment#standing-up-a-new-environment) and `npm run bootstrap:env` — it sets all five secrets from `scripts/environments/<name>.env` in one pass after you fill the template.

Local stack only: `supabase/functions/.env` + `supabase functions serve r2-grant r2-branding-upload`. Until provisioned: `CDN steps disabled — Storage grant refused (503)`.

Functions deploy with CI on push to `staging` / `main`.

### Fixing wrong URLs

Update env `R2_PUBLIC_DOMAIN`, then run the pipeline once so URL columns are rewritten.

**Symptom → cause cheat-sheet** (all seen in the wild):

- **Portal thumbnails blank, but the image opens fine in a new tab** — the CDN domain has Cloudflare **Hotlink Protection** on (blocks cross-site `<img>` by Referer). Turn it off for a public asset CDN; the portal also loads images with `referrerPolicy="no-referrer"` as a belt-and-braces.
- **Thumbnail requests return `200 text/html` (the app's `index.html`)** — the stored `thumbnail_url`s are **relative/hostless**, so the browser resolves them against the portal origin. Cause: the client synced with an **empty `r2_public_domain`** (URL became `"" + /thumbnails/…`). Fix: set the client's Public domain, **null the poisoned rows** (`update assets set thumbnail_url=null, download_url=null where client_id=…`) so the next run doesn't skip them, then run the pipeline.
- **CDN URL returns `403`** — the object isn't in the bucket (never uploaded, or wrong bucket). Re-run the pipeline with the correct `r2_bucket`.
- **Download saves a `.html` file** — same as the `text/html` case: `download_url` points at an HTML 404/error page, not the object.

The `update … set thumbnail_url=null` step matters because the pipeline pre-loads existing URLs from the DB and skips re-uploading anything it thinks is already there — stale URLs would otherwise make it skip forever.

### End-to-end rehearsal

Use **Staging Supabase + Staging R2** together ([Team staging setup](/getting-started/team-dev-setup)). Do **not** point a local database at a shared bucket — row URLs and objects will diverge across developers. Local DB without CDN is fine for schema/UI-only work.
