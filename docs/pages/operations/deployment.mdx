# Configuration and deployment

## Local development

Use the root launcher rather than opening three terminals manually:

```bash
npm run setup
npm run dev
```

The desktop uses fixed port 1420, docs uses 3001, and the web portal normally uses Vite's default 5173. See [Daily developer routine](/getting-started/development).

## Branches and environments

| Branch | Portal (Vercel) | Database + functions (CI) | Purpose |
|---|---|---|---|
| `dev` | ephemeral preview URLs | checks only (version, migration replay on PRs) | day-to-day integration; backend testing happens on the LOCAL stack |
| `staging` | `staging.hub.disruptcollective.com` (branch domain) | pushed to the **staging** Supabase project | rehearsal against hosted infra |
| `main` | `hub.disruptcollective.com` (production) | pushed to **production**, approval-gated | what clients and editors use |

Flow — **`main` is a promotion-only branch**: every pull request targets `staging` (from `dev` or feature branches); nothing PRs to `main` directly. After verifying on the staging infrastructure, a human opens the `staging → main` PR and merges it — that merge IS the release decision. Desktop releases ride `v*` tags on the merged `main` commit (`release-desktop.yml`). The local stack is not bound to any branch — it always runs whatever you have checked out.

Release ritual: bump the version on a branch (`npm run version:minor`, fill the changelog) → PR to `staging` → merge, test on staging → PR `staging → main` → merge → tag `vX.Y.Z` on the merge commit and push the tag.

Schema-dependent UI: merge the migration first (or in the same merge — migrations are additive-first by convention), since Vercel and the CI deploy in parallel on the same push.

### One-time platform checklist

**GitHub** (repo → Settings):
- Environments `staging` and `production`; on `production` enable **Required reviewers** (you).
- Secrets per environment: `SUPABASE_ACCESS_TOKEN` (both), `SUPABASE_STAGING_REF` + `SUPABASE_STAGING_DB_PASSWORD` (staging), `SUPABASE_PROD_REF` + `SUPABASE_PROD_DB_PASSWORD` (production).
- Branch protection on `main` (and optionally `staging`): require a pull request, block force pushes.

**Vercel** (project → Settings):
- Git → **Production Branch: `main`**.
- Domains → `hub.disruptcollective.com` → Production; `staging.hub.disruptcollective.com` → assign to branch `staging`. Create the CNAMEs Vercel shows in Cloudflare DNS.
- Environment Variables → `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY`: production values scoped to **Production**, staging values scoped to **Preview** (previews of every branch then run against staging — never against production data).

**Supabase Auth** (per project → Auth → URL Configuration):
- Production project: Site URL `https://hub.disruptcollective.com`; redirect list adds the staging domain is NOT needed here.
- Staging project: Site URL `https://staging.hub.disruptcollective.com`; add `https://*.vercel.app/**` to redirects so ephemeral previews can sign in.
- Both: keep `http://localhost:7623/auth-callback` for desktop sign-in.

**Supabase function secrets** (per project, once): `supabase secrets set CF_API_TOKEN=… CF_ACCOUNT_ID=… R2_PARENT_ACCESS_KEY_ID=…` — see [Database environments](/operations/database-environments#provisioning-the-storage-control-api-one-time-per-environment).

## Standing up a new environment

A hosted environment — a new tier (staging) or a whole new agency instance — splits into a **scriptable** half and a small **manual** half. `scripts/bootstrap-env.mjs` (`npm run bootstrap:env`) does the scriptable half in one idempotent, re-runnable pass:

- links the project and pushes the schema (all migrations)
- deploys edge functions
- sets edge-function secrets from `supabase/functions/.env` (the Cloudflare grant credentials)
- sets the Auth **Site URL + redirect allow-list via the Management API** — config as code, so the `localhost:3000` magic-link drift can't recur
- **invites the founding admin** and stamps `role=admin` (no chicken-and-egg, no hand-SQL — they click the email and they're in)
- optionally creates the first client with its storage config and assigns the admin to it

```bash
# 1. copy the template, then fill it in IN YOUR EDITOR (no shell escaping, ever)
cp scripts/environments/example.env.template scripts/environments/staging.env

# 2. preview — changes nothing
node scripts/bootstrap-env.mjs staging

# 3. execute
node scripts/bootstrap-env.mjs staging --execute
```

The config file (`scripts/environments/<name>.env`, gitignored) holds every value — project ref, site URL, admin email, the three Supabase secrets (access token = your account PAT; DB password + service key from the target project's dashboard), and the Cloudflare grant creds. You never type a secret or a flag on the command line, so there is nothing for the shell to mangle. Dry-run is the **default**; `--execute` is required to apply, and a production project ref is refused unless you add `--i-know-its-prod`. Buckets are **one per environment** (`dc-hub-staging`, `dc-hub-bucket`, …); a new agency gets its own config file and its own everything.

**The manual half** (platform-account actions the script deliberately doesn't automate — it prints this list at the end):

1. Create the Supabase project (dashboard, or `supabase projects create`).
2. Cloudflare R2: create the bucket, enable a public domain (r2.dev or custom), add CORS `GET`/`HEAD` from `*`; create an account-scoped R2 API token and put its Token value + Access Key ID into `supabase/functions/.env` before running (or re-run after).
3. Vercel: point a project/branch at the site URL with this environment's URL + anon key.
4. DNS for any custom domain.

After the script runs and the manual list is done, an editor adds the environment in the desktop (Settings → Environments) and signs in. Nothing else is hand-assembled.

## Portal hosting (Vercel)

`web/vercel.json` configures the build (workspace build command, `apps/client-hub/dist` output, SPA rewrites so client-side routes survive refresh). One-time setup:

1. Vercel → **Add New Project** → import the `mindb` repo → set **Root Directory** to `web` (the rest is read from `vercel.json`).
2. Environment variables: `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` (the production **publishable** key).
3. **Domains** → add the custom domain, create the CNAME it asks for in Cloudflare DNS.
4. Supabase (production) → Auth → URL Configuration: set **Site URL** to the portal domain and add it to the redirect list — portal magic links land there.

From then on every push to `main` redeploys the portal automatically, and pull requests get preview URLs. No GitHub Action is involved; Vercel's git integration is the pipeline.

## Desktop releases

A version tag (`vX.Y.Z`, created by the [versioning workflow](/reference/versioning)) triggers `release-desktop.yml`: a macOS runner builds the Tauri bundle and attaches the `.dmg` to a **draft** GitHub Release — review, add notes, publish. Builds are currently **unsigned**: on first launch editors must right-click → Open (Gatekeeper). Signing/notarization requires an Apple Developer ID and is the planned next hardening step.

Editor onboarding after install: pick/enter the environment (URL + anon key — public values), sign in with the magic link, done. Storage credentials are granted per run by the Control API; no per-machine CDN or key setup exists.

## Web and Supabase

The web app reads Supabase URL/anon key from environment (Vercel env vars in production) or portable localStorage settings. Configure Supabase Auth Site URL and allowed redirect patterns for every production/dev origin, including `http://localhost:7623/auth-callback` for desktop sign-in.

Desktop workstations need filesystem permissions plus native dependencies for requested thumbnail formats. Port 7623 must be available for OAuth and sign-in callbacks. OneDrive device flow needs outbound HTTPS but no callback listener.

R2 storage is server-authoritative: bucket and public domain live on the client row; credentials are minted per run by the `r2-grant` function (see [Database environments](/operations/database-environments#provisioning-the-storage-control-api-one-time-per-environment)). Deploy database migrations/RLS/RPC changes before UI features that depend on them, and regenerate database types when schema changes.

## Release artifacts

`npm run build` validates/builds the three JavaScript parts but does not produce signed desktop installers or deploy a hosted environment. Build Tauri bundles from `desktop` with `npm run tauri build`; hosting and signing credentials remain environment-specific and are not automated in this repository today.

All deployed parts should come from the same `vX.Y.Z` commit. Follow [Versioning and releases](/reference/versioning) rather than changing component manifests independently.
