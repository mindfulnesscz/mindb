# DC Hub documentation

DC Hub is a local-to-cloud digital asset publishing system. The **desktop application** (editor/admin signed-in) discovers and transforms files, publishes binaries to R2 and optional cloud drives, and synchronizes metadata to Supabase. The **web portal** reads that metadata and presents role-aware galleries, feedback, downloads, and administration (clients, users, tags, branding, export destinations).

These pages describe what the current source code actually does, including ordering, skip rules, failure boundaries, and known limitations.

## End-to-end data flow

1. One scan finds publishable files under configured `OUT` folders.
2. Optional local stages create thumbnails, collect package files, mirror local targets, export to cloud providers, and generate Obsidian notes.
3. R2 (when the environment’s `r2-grant` succeeds) stores browser-facing thumbnails and originals under `{client_id}/…`. Content hashes skip unchanged bytes.
4. Supabase receives asset rows, version history, CDN URLs, and cloud links. Tags, destination definitions, and client identity are managed on the web (DB source of truth); desktop keeps OAuth tokens local.
5. The web app queries Supabase, filters by client/status/role, and records comments, ratings, views, and downloads.

**Local generation, R2 upload, provider upload, and database synchronization are independent phases.** A green thumbnail line does not mean R2 or Supabase succeeded. See [How to read pipeline logs](/desktop/logs).
