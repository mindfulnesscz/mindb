# Supabase synchronization

The desktop uses the service-role key through REST/native request helpers, resolves/creates a client row, parses scanned filenames, and upserts current assets. It also synchronizes taxonomy and version history. CDN and provider URL maps produced earlier in the run are attached to corresponding records.

Legacy identity matches by shortcode and can delete stale rows. Migrated identity matches stable/child IDs, preserves rename continuity, and marks missing records disconnected. Targeted R2 cleanup protects object keys still referenced by active rows, particularly shared/gallery thumbnail cases.

Synchronization is not an atomic transaction with filesystem, R2, or provider APIs. A retry must reconcile partial state. Service-role access also means a bug can bypass normal RLS, so test new synchronization changes on a non-production client and keep backups.
