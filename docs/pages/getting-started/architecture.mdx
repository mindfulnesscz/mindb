# Architecture

DC Hub is one Git monorepo and one coordinated product release. The root contains developer orchestration, versioning, CI, and the shared changelog; `desktop`, `web`, and `docs` retain independent dependency trees and lockfiles.

```text
root scripts / CI / version / changelog
├── desktop: authenticated publisher (editor/admin)
├── web: browser portal (anon + session under RLS)
├── supabase: migrations, seed, edge functions (r2-grant, branding)
└── docs: contributor and operator handbook
```

Capability ownership (who edits which fields) is documented in [Platform division](/getting-started/platform-division). Day-to-day team work uses [shared Staging](/getting-started/team-dev-setup).

## Components and trust boundaries

| Component | Runs where | Credentials | Responsibility |
|---|---|---|---|
| Desktop React UI | Tauri webview | User session (PKCE magic link) | Pipeline, settings, destinations, logs |
| Desktop Rust backend | Native process | Short-lived R2 grant per run; OAuth tokens | Thumbnails, R2 signing, provider uploads |
| Supabase | Postgres / Auth / PostgREST / Functions | Desktop/browser: anon + user JWT; edge: service secrets | Metadata, profiles, roles, RLS, grants |
| Cloudflare R2 | Object storage / CDN | Environment secrets on edge functions only | Public thumbnails, originals, logos |
| Dropbox / OneDrive / Drive | Provider APIs | Machine-local OAuth tokens | Optional delivery copies |
| Web portal | Browser | Anon key + user session | Galleries, downloads, admin (clients, users, tags) |

The desktop signs in as a staff user (**editor** or **admin**). Pipeline writes go through that session under RLS. Permanent Supabase service-role and parent R2 keys never live in the desktop — R2 access is minted by `r2-grant`. The web app must only receive the anon/publishable key.

Remaining residual risk: cloud **OAuth tokens** are still machine-local plaintext until protected storage lands. See [Auth & secrets roadmap](/desktop/authentication-plan).

## Control flow

`PipelineView` builds a run context from the active client, settings, vocabulary, destinations, and a storage grant when CDN is enabled. `runPipeline` scans once and runs stages in fixed order. After it returns, the UI syncs assets and version history to Supabase. Tags are administered on the web (DB source of truth); the desktop may process `rename_tasks` after tag changes.
