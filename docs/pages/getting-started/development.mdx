# Daily developer routine

This page is the fastest path from a fresh clone to a reviewable contribution. DC Hub is one Git monorepo with three runnable applications and one shared release version.

For who owns which fields, see [Platform division](/getting-started/platform-division). For Staging as the shared backend, see [Team staging setup](/getting-started/team-dev-setup).

## 1. Install prerequisites

| Tool | Why it is needed |
|---|---|
| Node.js 22 and npm | Root launcher, desktop/web builds, documentation |
| Rust stable (`rustc`, `cargo`) | Tauri native application and Rust checks |
| Tauri 2 platform prerequisites | Native window/WebView and application bundles |
| Git | Source control and Nextra edit metadata |
| Docker | Local Supabase for migration work only |

Optional for thumbnail conversion: LibreOffice, Poppler (`pdftoppm`), WebP (`cwebp`).

## 2. Clone and install

```bash
git clone <repository-url> dc-hub
cd dc-hub
npm run setup
```

There is intentionally no root dependency tree. `npm run setup` runs `npm ci` in `desktop`, `web`, and `docs`. Rerun after lockfile changes.

## 3. Configure backends

**Preferred for feature work:** shared Staging — fill `web/apps/client-hub/.env.staging`, then `npm run dev:web:staging`. In the desktop, add a Staging environment (URL + anon key) and sign in as admin/editor.

**Local schema work only:**

```bash
cp web/apps/client-hub/.env.local.example web/apps/client-hub/.env.local
# VITE_SUPABASE_URL=http://127.0.0.1:54321
# VITE_SUPABASE_ANON_KEY=<local anon from supabase status>
npm run db:start
npm run db:reset
```

Only the anon/publishable key belongs in browser/desktop config. Do **not** put the service-role key or permanent R2 keys in the desktop. Folders and OAuth tokens stay machine-local; CDN uses `r2-grant` when Staging secrets are provisioned.

Never mix local Supabase with a shared R2 bucket for pipeline work.

## 4. Start the system

```bash
npm run dev
```

| Label | Command | Address |
|---|---|---|
| `desktop` | Tauri | native window; Vite `localhost:1420` |
| `web` | client-hub | normally `localhost:5173` (default `.env.local`) |
| `docs` | Nextra | `localhost:3001` |

Focused starts: `npm run dev:desktop`, `npm run dev:web`, `npm run dev:web:staging`, `npm run dev:docs`. Port `7623` must be free for Dropbox/Google OAuth and desktop magic-link callbacks.

## 5. Choose the correct change boundary

| Change | Start here | Also review |
|---|---|---|
| Desktop screen / pipeline | `desktop/src/features`, `services` | CDN grant, identity, sync docs |
| Native I/O | `desktop/src-tauri/src` | Tauri capabilities, `check:rust` |
| Naming / filename tags | `desktop/src/domain` | Vocabulary for **filenames**; DB tags admin on web |
| Portal UI / admin | `web/apps/client-hub/src/features` | Platform division, permissions |
| Schema / RLS / RPC | `supabase/migrations` | `db:types` → `@dc-hub/database` |
| Shared portal rules | `web/packages/asset-library` | Roles: public / member / editor / admin |

Desktop is staff-session privileged under RLS; the web stays anon + session. UI helpers do not replace RLS.

## 6. Feedback loop

1. Pull latest base branch; keep unrelated changes out.
2. Prefer Staging for end-to-end; local only for migrations.
3. Make one coherent change; cover success + empty/failure paths.
4. Update docs when behavior, settings, schema, or operator steps change.
5. Inspect `git diff` for secrets and accidental lockfile churn.

Do not run two pipelines against the same Staging client/source tree at once.

## 7. Testing by area

```bash
npm run check          # version + desktop build + web typecheck + docs + rust
npm test               # desktop Vitest (filename parsing and growing suites)
```

CI (`.github/workflows/check.yml`) runs the same on pull requests.

### Database changes

```bash
supabase migration new <name>
npm run db:reset
npm run db:types       # writes web/packages/database/src/database.types.ts
```

### Documentation

```bash
npm run build:docs
npm run dev:docs
```

## Before opening a pull request

```bash
npm run check
npm test
git status --short
```

Describe the boundary, manual scenarios, schema impact, and screenshots (no client secrets).

## Releasing

```bash
npm run version:patch   # or minor / major
```

See [Versioning and releases](/reference/versioning). Tag `vX.Y.Z` for desktop release CI.

- [ ] `npm run setup` and `npm run dev` (or Staging web + desktop Staging env)

- [ ] Never put service-role or parent R2 keys in the desktop
- [ ] Desktop login required (editor/admin); residual OAuth storage is still local — [roadmap](/desktop/authentication-plan)
